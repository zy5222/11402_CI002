# UVa 540 - Team Queue

### 1. 題目摘要

有 $T$ 個隊伍，每個隊伍有多名隊員。現在大家要排隊，排隊規則如下：
* 當一個人進入排隊隊伍時，如果隊伍中**已經有他的同隊隊友**，他就可以直接「插隊」排在所有隊友的最後面。
* 如果隊伍中**沒有任何他的隊友**，他只能乖乖排在整條隊伍的最尾端。
給定每個人的隊伍歸屬，並執行多筆 `ENQUEUE` (進入佇列) 與 `DEQUEUE` (離開佇列並輸出) 操作，直到 `STOP` 為止。

### 2. 解題邏輯

這題無法只用單一一個 Queue 來解決，因為會發生插隊的狀況。我們需要使用**「雙層佇列 (Queue of Queues)」**的概念來實作：

1. **查表建立**：首先用一個陣列 `team_of` 記錄「哪個人屬於哪個隊伍」。
2. **主佇列 (`main_queue`)**：用來記錄「目前排隊中的隊伍順序」（裡面存的是隊伍 ID）。
3. **分隊佇列 (`team_queues`)**：針對每個隊伍，開一個專屬的佇列，用來記錄「該隊伍目前在排隊的成員」（裡面存的是人員 ID）。
4. **狀態陣列 (`in_queue`)**：用來快速判斷某個隊伍目前是否已經有人在主隊伍裡排隊。

**模擬操作：**
* **ENQUEUE**：查詢這個人的隊伍 ID。如果他的隊伍不在 `main_queue` 裡，就把隊伍 ID 推進 `main_queue`，並標記為在隊伍中。接著，把這個人推進該隊的 `team_queues` 裡。
* **DEQUEUE**：看 `main_queue` 最前面的隊伍是誰，從該隊伍的 `team_queues` 中彈出最前面的人並輸出。如果該隊伍的人都彈出光了（變成空的），就把這個隊伍 ID 從 `main_queue` 中彈出，並將狀態改回不在隊伍中。

### 3. 筆記

* **時間複雜度**：
  雖然排隊邏輯看起來有些複雜，但因為我們巧妙地利用了多個 Queue，不管是找隊友（陣列 $O(1)$）、插隊（推入分隊陣列 $O(1)$）還是出列（從最前面彈出 $O(1)$），所有的操作時間複雜度都維持在 $O(1)$。
* **陣列取代 Map 的效能優化**：
  題目有提到人的 ID 最大只到 $999999$，所以在儲存「人員 ID 到隊伍 ID」的對應時，直接開一個長度為 $10^6$ 的一維陣列來當作 Hash Table 會比使用 `std::map` 或 `std::unordered_map` 快上非常多。
* **輸出格式**：
  每一輪 Scenario 結束後都要輸出一個空白行，這是 UVa 經典的排版要求。
