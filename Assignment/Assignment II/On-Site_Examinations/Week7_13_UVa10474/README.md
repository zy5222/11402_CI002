# UVa 10474 - Where is the Marble?

### 1. 題目摘要

這題的情境是找大理石上的數字。給定 `n` 顆大理石（上面有數字）和 `q` 次查詢。
規則有幾個重點：
1. **先排序**：在回答查詢前，必須先把大理石上的數字「由小到大」排好。
2. **找最左邊的**：如果有重複的數字，要回答最先出現（最左邊）的那一個位置。
3. **輸出格式**：有找到就印出 `found at [位置]`，沒找到就印 `not found`。

### 2. 我的解題邏輯

這題的架構其實很直觀，主要是對陣列做操作跟搜尋。

* **讀取與排序**：我開了一個 `vector<int> v(n)` 來存所有大理石的數字。讀完測資之後，直接套用 `<algorithm>` 裡的 `sort(v.begin(), v.end())` 把數字從小到大排好，這是這題最核心的前提。
* **線性搜尋**：在處理 `q` 次查詢時，我跑一個 `for` 迴圈從頭遍歷陣列。因為題目要求找「最先出現」的，所以從左邊開始找，只要一遇到目標數字就印出答案。
* **開關機制 (Flag)**：為了解決「什麼時候該印 not found」的問題，我在內層迴圈外設了一個 `bool found = false`。如果有找到就切換成 `true` 然後馬上 `break` 跳出；如果跑完迴圈 `found` 依然是 `false`，就代表陣列裡根本沒這個數字，這時才印出 `not found`。

### 3. 筆記與踩坑紀錄

* **電腦與人類的數法差異（索引陷阱）**：
  這是我一開始覺得很怪的地方。陣列是從 `0` 開始存的（第一格是 `v[0]`），但題目要求的答案位置是從 `1` 開始算。
  我本來想說乾脆把迴圈寫成 `for (int j = 1; ...)`，結果發現這樣會直接漏看陣列裡的第一個數字！**正確的做法是妥協：搜尋迴圈還是得乖乖從 `0` 開始跑，只有在最後 `cout` 印答案的那一刻，才把 `j + 1` 轉成題目要求的位置格式。**
* **忘記排序直接 WA**：
  題目敘述裡有提到主角會先把數字排好。如果沒先 `sort` 過，找出來的位置絕對跟答案對不上。
* **未來 CPE 的優化方向**：
  我這個是用單層 `for` 迴圈從頭慢慢找。因為陣列已經排好序了，如果之後遇到測資幾萬筆的狀況，這段很有可能會超時。之後會試著把找數字這段改成 C++ 內建的 `lower_bound`（二分搜尋法），速度會快很多。
